# Project assessment vs. CC-06 Buc-ee's MVP v1.2

**Date:** 2026-05-08
**Source of truth (target):** [`docs/canonical/CC-06_Bucees_MVP_v1.2.md`](../canonical/CC-06_Bucees_MVP_v1.2.md)
**Source of truth (current):** repo at HEAD on `main` (commit `ff99128` and predecessors)
**Author:** Engineering snapshot
**Scope:** Code + schema + seed state. Excludes content authoring (campaign markdown, copy review) which is governed by SPEC-12 and CatalogLoader.

---

## Executive summary

The platform foundation is in good shape. The two-tier `Tenant → Location` collapse landed cleanly; users, locations, invitations, proposals, campaigns, and the campaign-execution loop (sweep + reply poll) are all wired up and exercised by 647 passing tests. The sidebar, nav, role-gating, profile, jobs index, and proposal edit/show have been brought in line with the operator UX described in the MVP.

**Where the project meets MVP scope:** wedge criteria #2, #3, and #4 (proposal-in / follow-up / stop / Needs Attention) are all functional end to end. Foundation criteria #5 (originator identity, location data) and #6 (template authoring 5/17) are largely complete in the schema and seed catalog.

**The two material gaps are wedge #1 (DASH job number) and foundation #7 (Analytics MTD/YTD + branch comparison).** Without #1 we don't ship to Jeff; without #7 the demo loses its recruiting payload under CC-07.

The Admin Portal posture is *inverted* from what v1.2 calls for: the spec asks for a backend-only sliver; the project has an extensive Platform Admin **frontend** but is missing the Account-level data fields (`logo_url`, `company_name`) that signature composition depends on per SPEC-07 v1.4. The frontend work is not wasted (it accelerates Milestone 2) but the data-layer omission is a go-live blocker.

### #5 — Basic credibility (Account/Location data) — **Partial; data-field gap**

| Item | State | Where |
|---|---|---|
| Attachments work | ✅ | `JobProposalAttachment` + Active Storage (`uploaded_by_user_id`, file blob). |
| Originator identity in salutation/signature | ✅ | `MailGenerator#substitute` resolves `{originator_*}` from `JobProposal.owner` (first/last/phone/email). Signature appended after the RFC 3676 delimiter. |
| Email quality consistent | ⚠️ | Signature drops missing pieces gracefully (verified by `mail_generator_test`). Acceptable for pilot. |
| Account `logo_url` populated | ❌ Missing | `tenants` table has only `name`. Spec calls for `logo_url`. SPEC-07 v1.4 §9 says signature composition fails without it. |
| Account `company_name` populated | ⚠️ | `tenants.name` is doing double duty as company name. `MailGenerator` resolves `{company_name}` from `tenant.name` (correct fallback) but the spec wants a distinct `company_name` field separable from the tenant's identity label. |
| Location signature-bearing fields | ✅ | `display_name`, `address_line_1/2`, `city`, `state` (2-letter), `postal_code`, `phone_number` all NOT NULL on `locations`. |
| Location activation gated on field presence | ⚠️ | Presence validators are unconditional, so a location without those fields can't save at all — effectively `is_active=true` requires them. The spec wants an explicit "blocked until populated" check tied to `is_active`. Functionally equivalent today; semantically softer than the spec. |
| Three Servpro locations seeded | ❌ Pilot data | Demo seed creates two locations under "Demo Roofing Co." (Naperville HQ, Rockford Branch). Jeff's three Servpro locations (NE Dallas, Boise, Reno) aren't seeded — they're an SMAI ops task per §K, not a code task, but worth flagging. |

**Action:** add `tenants.logo_url`, `tenants.company_name` (or rename `name` semantically and add `display_name` if needed); update `MailGenerator` to resolve from these. The Account fields are the only hard schema gap blocking signature correctness for v1.4.

| Item | State | Where |
|---|---|---|

## Schema-level gap summary

Concrete columns/tables to add, in priority order:

1. **`tenants.logo_url`** (string) and **`tenants.company_name`** (string, nullable; falls back to `tenants.name`) — unblocks SPEC-07 v1.4 signature composition.
2. **`tenants.job_reference_required`** (boolean, default false) — gates DASH-number requirement per criterion #1.
3. **`job_proposals.dash_job_number`** (string, nullable; required when the tenant flag is on) — captures Jeff's primary tracking ID.
4. **`users.title`** (string, nullable) — SPEC-07 v1.4 §H.
5. **`application_mailboxes.location_id`** (FK, nullable until backfill) — PRD-09 §I one-per-location.
6. **`audit_logs`** (table; tenant_id, actor_user_id, action, target_type, target_id, payload jsonb, created_at) — PRD-10 v1.3.1 requirement under §K.
7. **`job_proposals.template_version_id`** or equivalent — PRD-01 v1.4.1; supports §E "show resolved template version" and SPEC-11 v2.0.2 variant activation.

## Recommended next-up sequence

The MVP's priority-ordered Definition of Done puts the wedge first. The smallest set of changes that closes the wedge gap:

1. **DASH job number end-to-end** (criterion #1). Add the tenant flag + the proposal field + the prompt in the new-job UI (or a required-on-edit gate) + the subject-line prefix in `MailGenerator`. This is a one-PR slice.
2. **Account `logo_url` + `company_name`** (criterion #5). Tiny migration, two MailGenerator branches, a one-time data seed for Jeff. Unblocks signature correctness.
3. **`users.title`** (criterion #5 / §H). Add column, expose on profile + invite form, render in salutation.
4. **One mailbox per location** (criterion #5 / §I). Migration on `application_mailboxes` + a small refactor in `ApplicationMailbox.current`.

After the wedge, the foundation:

5. **MTD/YTD on the Conversion Rate tile** (SPEC-05 v1.0). Backend ships raw aggregations; frontend buckets. Wave 6D architecture.
6. **Branch comparison view** (SPEC-06 v1.0). Per-location breakdown row under the Conversion Rate tile.
7. **`template_version_id` on JobProposal + show-page surface** (PRD-01 v1.4.1, §E).
8. **Audit logging on Admin Portal writes** (PRD-10 v1.3.1).

If any of #1–#4 slip past the wedge, the pilot is at risk per the spec's own framing. Items #5–#8 carry the platform thesis to the Stone Family Office Design Partnership recruitment — the foundation deliverable that proves this is more than Jeff-specific automation.

## Caveats

- This assessment reads code, schema, and seed data only. It does not validate operational readiness (DNS for `mail` subdomains, OAuth credentials in production, Heroku add-ons, monitoring) or content quality of the 17 authored campaigns.
- "Wave 6D Analytics architecture" guidance was inferred from the v1.2 doc; specific contract surfaces should be cross-checked against PRD-07 v1.2 §1A, SPEC-05 §1A, SPEC-06 §1A before any implementation lands.
- The `Development Job Type` row in the database is a dev artifact — production seed data shouldn't carry it forward. Worth a cleanup or a `is_internal` flag if it's load-bearing for tests.
- Several PRDs and SPECs in the canonical project reference behaviors that don't yet have a build trail (e.g. the "two-step atomic activation" in SPEC-11 v2.0.2). The "Complete MVP build set" in v1.2 makes scope explicit; this report focuses on the items v1.2 itself raises in its success-criteria text.
