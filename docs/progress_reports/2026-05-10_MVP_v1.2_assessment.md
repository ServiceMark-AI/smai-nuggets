## Executive summary

**The single largest cross-PRD contract gap from the prior report — append-only proposal history — is now closed.** `paper_trail` was adopted with a locked-down `Version` subclass that raises on update / destroy, the JobProposal model is wired with `has_paper_trail`, and the proposal show page now carries an Activity card with a per-version detail page. That single change retires the gap that blocked PRD-01 §12, PRD-04 §9 ("Handled by SMAI" feed), PRD-06 §10 (Activity Timeline), and SPEC-09 §9 (loss-reason history-event payload).

**SPEC-11 § Campaign Template Architecture moved from "footing only" to "substantial."** A `campaign_revisions` table landed with a `drafting / active / retired` status enum, partial unique index that DB-level enforces only-one-active-per-campaign, and `created_by_user` / `approved_by_user` audit fields. The Campaign show page now exposes a Revisions table; `Admin::CampaignRevisionsController#approve` retires the prior active revision in the same transaction it activates a new one. Step editing is gated on a draft revision. The atomicity, immutability, and authoring metadata that SPEC-11 §11 calls for are largely in place; the remaining gaps are `authoring_hypothesis`, `industry_classification`, and `deactivated_at`.

**Loss-reason capture went from free-text to a constrained taxonomy.** A `loss_reasons` table with six product-defined codes + a FK on `job_proposals` replaced the prior string column, the Mark Lost modal renders a dropdown, the proposal show page surfaces the picked reason in a "Why we lost this job" card, and the analytics dashboard rolls up lost jobs by reason in a CSS conic-gradient pie. SPEC-09's history-event payload is now meaningful (the change rides paper_trail).

**Net v1.2 readiness:** all four wedge criteria still hold. Foundation criteria #5 and #6 still complete. Foundation criterion #7 (Analytics) added a **Why we lost** breakdown — but the engagement-side metrics (Avg Time to First Reply, funnel stages 3–4, originator reply-rate column) still depend on the inbound-message data path that hasn't shipped.

**Notable infra change:** error reporting moved from Bugsnag to Sentry (PR #179). The bundled fallback DSN is the only thing that ships in code; production overrides via `SENTRY_DSN`. Browser-side error tracking is dropped pending a separate JS-SDK PR.

8. **`application_mailboxes.location_id`** — PRD-09 §I one-per-location. Still missing.
9. ~~`job_proposal_history`~~ ✅ **Closed via `paper_trail`.** The implementation uses paper_trail's polymorphic `versions` table rather than a dedicated table; the contract is met (append-only, immutable, actor + payload), but only `JobProposal` is currently wired with `has_paper_trail`. Other models (CampaignInstance, CampaignStepInstance, CampaignRevision activate / retire) would need their own opt-in if their state changes need surfacing in the timeline.
10. **`scenarios.industry_classification`** — SPEC-03 §13.2 author-facing metadata. Still missing.
11. **Authoring metadata on `campaigns` per SPEC-11 §11** — `is_active`, `authored_by`, `authored_at`, `activated_at`, `deactivated_at`, `authoring_hypothesis` on `Campaign` itself. **Substantially closed via `campaign_revisions`** which carries `revision_number`, `status` (drafting/active/retired), `created_by_user`, `approved_by_user`, `approved_at`. **Still missing:** `authoring_hypothesis`, `industry_classification`, an explicit `deactivated_at`. The `campaigns.template_version_id` (string) column predates this work and is now arguably obsolete; SPEC-11's `template_version_id` semantics are better captured by `(campaign_id, revision_number)`.

**New table on this report:** `loss_reasons` (six product-defined codes with `display_name` + `sort_order`). Closes SPEC-09's "free text loss reason makes Analytics impossible" data-quality gap.

## Caveats

- This assessment reads code, schema, and seed data only against HEAD `b7f9810`. It does not validate operational readiness or content quality of the 17 authored campaigns.
- The four wedge criteria in CC-06 v1.2 §"MVP success criteria (non-negotiable)" are the binding constraint; this report's "Functional" markers on each remain the load-bearing finding.
- The `paper_trail`-based audit trail meets the contract but uses a generic `versions` table rather than a per-model `job_proposal_history` table. If future PRDs want denormalized per-event history queries, a materialized view could be built off `versions WHERE item_type = 'JobProposal'` rather than re-implementing the storage layer.
- The browser-side error-reporting regression introduced by the Bugsnag → Sentry swap is the only known regression in this delta. Not a v1.2 blocker, but worth pinning before the pilot demo.
